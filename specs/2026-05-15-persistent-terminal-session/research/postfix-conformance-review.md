## Safety model

The safety model is now honestly documented across the spec and user docs. The product behavior explicitly says the first version is a direct extension-process shell runner and does not provide the same per-command approval flow as pi's one-shot `bash` unless the host sandbox already constrains the extension process (`specs/2026-05-15-persistent-terminal-session/PRODUCT.md:21`). The README repeats that user-facing caveat (`README.md:30`), and TECH frames host-mediated persistent execution as a follow-up rather than current behavior (`specs/2026-05-15-persistent-terminal-session/TECH.md:115`).

That documentation matches the implementation: `exec_command` runs `child_process.spawn(command, { shell: true, stdio: "pipe", detached: process.platform !== "win32" })` directly in the extension process (`extensions/terminal-session.ts:371`). The tool prompt guidelines add a model-facing warning not to use it for destructive, privileged, or sensitive commands unless explicitly requested (`extensions/terminal-session.ts:312`), but this is guidance rather than enforcement.

Remaining safety implications are therefore transparent rather than hidden: shell metacharacters are interpreted by the platform shell, approvals are not inherited from built-in `bash`, cleanup through Ctrl-C is cooperative, and process-tree guarantees are platform-sensitive. This is acceptable only if release notes keep the same direct-runner caveat visible.

## Remaining blockers

No blocker remains from the earlier safety/documentation contradiction: the product, tech spec, and README now align with the direct extension-process runner that is implemented.

Potential release blocker, depending on strictness: Behavior #7 remains only partially proven. Invalid workdirs and pre-start cancellation are handled without `session_id`, but spawn/permission-denial and abort-after-spawn behavior are not tested. Because abort after process creation uses SIGTERM and waits only for the yield window, a stubborn process can plausibly remain running and produce a session despite cancellation. If Behavior #7 is a hard release contract, add cancellation/spawn regression tests and clarify or harden abort cleanup before release.

Acceptable follow-ups if the release is scoped as a documented first version:

- Add a host-mediated persistent process API so sessions can inherit pi's normal command approvals/sandboxing instead of using direct `spawn()`.
- Add a force-stop or explicit stop tool for commands that ignore Ctrl-C, plus Windows process-tree validation.
- Tighten Behavior #6 wording from `cleaned up` to `expired/cleaned up`, or emit an explicit `cleaned` tombstone reason when tombstone retention evicts old terminal states.
- Add workflow-level integration coverage for a real dev-server-style command and an independent check while the session remains alive.

## Verdict

Release status: acceptable with one explicit caveat, or blocked only if Behavior #7 cancellation/spawn semantics are considered mandatory before shipping.

The post-fix state is substantially better than the earlier reviews: PRODUCT.md Behaviors #1-#12 mostly match the implementation, tests, README, and testing docs; the high-output cursor issue has a regression test; tombstones make post-exit and post-interrupt cleanup observable; and the safety model is no longer misleading. The direct shell runner remains a real safety limitation, but it is now honestly specified and documented rather than hidden.

Recommended release note: ship as a non-PTY, process-local, direct extension-process shell runner for trusted agent workflows, with no built-in `bash` approval parity and cooperative Ctrl-C cleanup only.
