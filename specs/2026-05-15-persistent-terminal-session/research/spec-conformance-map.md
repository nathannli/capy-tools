# Spec Conformance Map: Persistent Terminal Session Tools

Date: 2026-05-15
Scope: `specs/2026-05-15-persistent-terminal-session/PRODUCT.md`, `TECH.md`, `extensions/terminal-session.ts`, `tests/terminal-session.test.ts`, `README.md`, and `docs/testing.md`.
Validation run: `bun test tests/terminal-session.test.ts` passed (9 tests, 0 failures).

## Overall Readiness Assessment

Status: **mostly implemented, not release-ready without either a safety clarification or a safety integration fix**.

The core persistent-session workflow is present: `exec_command` starts shell-backed sessions, returns structured details, maintains a module-level registry, supports polling and stdin writes through `write_stdin`, isolates multiple sessions, caps returned output, and renders compact one-line results. The focused terminal test suite passes.

Main gaps:

1. **Behavior #9 is contradicted or at least unproven.** The implementation calls `child_process.spawn(..., shell: true)` directly from the extension (`extensions/terminal-session.ts:305`) and does not delegate command execution to pi's existing `bash`/approval/sandbox path. Prompt guidance warns the model not to use destructive commands (`extensions/terminal-session.ts:247`), but that is not enforcement.
2. **Exited/stopped session state is retained too briefly for Behaviors #6 and #12.** Sessions are deleted immediately after a quick exit (`extensions/terminal-session.ts:327`) and after any `write_stdin` result observes exit (`extensions/terminal-session.ts:378`), so later polls often receive the generic "never created or already cleaned up" error (`extensions/terminal-session.ts:353`) rather than a precise already-exited or cleaned-up terminal state.
3. **Behavior #7 is only partially covered.** Invalid workdirs and pre-start cancellation are handled, but spawn errors, permission denials, and abort-after-spawn behavior are not tested; abort-after-spawn can still return a running `session_id` if SIGTERM does not complete before the wait window.
4. **Behavior #10 is under-tested and partially weaker than specified.** Tests cover only collapsed/expanded rendering for a running `exec_command`; they do not cover `write_stdin`, exited, empty-output, or error render states requested by `TECH.md`.
5. **README/testing docs describe the happy path accurately but omit important shipped limits.** They do not mention non-PTY-only behavior, process-local lifetime, byte caps/truncation metadata, or the unresolved safety/approval caveat.

## README and Testing Docs Accuracy

- `README.md` accurately lists `terminal-session` and the `exec_command`/`write_stdin` tools (`README.md:14`) and correctly describes exit-vs-session return behavior and empty-poll/Ctrl-C usage (`README.md:28`, `README.md:30`).
- `README.md` does not mention non-PTY-only behavior, process-local session lifetime, `max_output_bytes`, truncation metadata, or the safety/approval uncertainty. These are important shipped semantics from `PRODUCT.md:39`, `PRODUCT.md:40`, and `PRODUCT.md:41`.
- `README.md` testing coverage statement includes `exec_command`/`write_stdin` (`README.md:109`) and is broadly accurate for the focused suite.
- `docs/testing.md` coverage map says terminal tests exercise interruption (`docs/testing.md:55`), but interruption is only used by the `stopSession()` cleanup helper (`tests/terminal-session.test.ts:22`) and has no assertion that SIGINT cleanup or post-stop terminal-state reporting works.
- `docs/testing.md` recommended workflow remains accurate and encourages adding focused regression tests (`docs/testing.md:71`).

## Minimal Additional Tests

1. Add a `write_stdin` Ctrl-C stop test that asserts the interrupt result reports `status: exited` or `signal`, then decide whether a subsequent poll should return a tombstone terminal state or a cleaned-up error.
2. Add already-exited/cleaned-up id tests for Behavior #6 so unknown, exited, and cleaned states are distinguishable, or update PRODUCT.md to accept the current generic cleaned/unknown error.
3. Add abort tests: pre-start abort, abort after spawn, and an assertion that no `session_id` is returned when cancellation wins.
4. Add spawn/permission-denial tests that document the intended shell-backed behavior for nonexistent commands and non-executable files.
5. Add renderer tests for `write_stdin`, exited, empty-output, truncation, and error states.
6. Add a safety conformance test or integration note only after deciding how pi host approvals/sandboxing are supposed to apply to extension-spawned shell commands.

## Minimal Spec or Documentation Edits

- If direct `spawn()` remains the implementation, edit `PRODUCT.md` Behavior #9 and README to explicitly state that `exec_command` is a raw extension shell runner and does not currently integrate with built-in `bash` approval/sandbox semantics.
- If one-shot terminal-state observability is intended, edit Behavior #12 to say the terminal state is observable on the first poll/write that notices exit; otherwise implement retained tombstones for recently exited/stopped sessions.
- Add README notes for `tty: true` unsupported, process-local session lifetime, Ctrl-C cleanup, output caps, and truncation metadata.
- Adjust `docs/testing.md` to say interruption is used for cleanup unless a stop assertion test is added.
